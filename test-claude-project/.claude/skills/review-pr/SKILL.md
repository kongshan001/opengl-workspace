# PR 代码审查

审查指定的 Pull Request，提供详细的代码审查报告。

## 使用方法

```bash
# 审查指定 PR
/review-pr 123

# 审查当前分支变更（未创建 PR 时）
/review-pr
```

## 审查流程

1. **获取变更**
   - 使用 `gh pr diff` 获取 PR 变更
   - 或使用 `git diff main` 获取本地变更

2. **逐文件审查**
   - 分析每个变更文件
   - 识别关键改动

3. **多维度检查**
   - 代码质量和可读性
   - 潜在 Bug 和逻辑错误
   - 安全漏洞
   - 性能问题
   - 测试覆盖率
   - 文档完整性

4. **生成报告**
   - 按严重程度分类问题
   - 提供具体改进建议

## 审查标准

### 代码质量

- [ ] 代码是否清晰易懂
- [ ] 命名是否有意义
- [ ] 函数是否过长（> 50 行警告）
- [ ] 是否有重复代码
- [ ] 是否遵循项目编码规范

### 安全检查

- [ ] 输入是否经过验证
- [ ] 敏感数据是否正确处理
- [ ] 是否有 SQL 注入风险
- [ ] 是否有 XSS 风险
- [ ] 认证和授权是否正确

### 性能

- [ ] 是否有 N+1 查询问题
- [ ] 是否有不必要的循环
- [ ] 数据库查询是否优化

### 测试

- [ ] 新代码是否有测试
- [ ] 边界条件是否覆盖
- [ ] 测试是否有意义

## 输出格式

```markdown
## PR 审查报告

**PR**: #<number> - <title>
**审查时间**: <date>

### 📊 总体评价

<overall assessment>

### 🔴 严重问题 (必须修复)

1. **[文件:行号]** 问题描述
   ```diff
   - 问题代码
   + 建议修改
   ```

### 🟡 建议改进 (推荐修复)

1. **[文件:行号]** 建议描述

### 🔵 小问题 (可选修复)

1. **[文件:行号]** 小问题描述

### ✨ 亮点

- 优秀的实现点

### 📝 建议

1. 具体建议
2. ...

### 🏁 结论

[✅ Approve / 🔧 Request Changes / 💬 Comment]
```

## 注意事项

- 保持客观和建设性
- 提供具体的改进建议
- 解释为什么需要修改
- 认可优秀的代码
