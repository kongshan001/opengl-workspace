# Self-Improving Skill 调研文档

## 一、技能概述

### 1.1 核心理念
Self-Improving 是一个让 AI Agent 具备**自我反思**和**持续学习**能力的技能。它通过：
- 记录用户的纠正和反馈
- 自动识别重复模式
- 将学习成果持久化存储
- 在后续交互中应用已学知识

### 1.2 设计哲学
```
"Knowledge should compound over time without manual maintenance."
（知识应该随时间复利增长，无需人工维护）
```

## 二、架构设计

### 2.1 分层记忆系统

```
~/self-improving/
├── memory.md          # 🔥 HOT: ≤100行，始终加载
├── index.md           # 主题索引，行数统计
├── corrections.md     # 纠正日志（最近50条）
├── projects/          # 🌡️ WARM: 项目级学习
│   ├── my-app.md
│   └── side-project.md
├── domains/           # 🌡️ WARM: 领域级学习
│   ├── code.md
│   ├── writing.md
│   └── comms.md
└── archive/           # ❄️ COLD: 归档模式
    └── 2025.md
```

### 2.2 三层热度机制

| 层级 | 位置 | 大小限制 | 加载策略 |
|------|------|----------|----------|
| HOT | memory.md | ≤100行 | 始终加载 |
| WARM | projects/, domains/ | ≤200行/文件 | 按需加载（上下文匹配时） |
| COLD | archive/ | 无限制 | 显式查询时加载 |

### 2.3 命名空间继承

```
global (memory.md)
  └── domain (domains/code.md)
       └── project (projects/app.md)
```

**冲突解决规则**：
1. 最具体的优先（project > domain > global）
2. 同层级最新优先
3. 模糊时询问用户

## 三、学习机制

### 3.1 触发条件

| 触发语 | 置信度 | 动作 |
|--------|--------|------|
| "不对，应该是X" | 高 | 立即记录纠正 |
| "我之前说过..." | 高 | 标记为重复，提升优先级 |
| "总是/永远不要做X" | 确认 | 直接提升为偏好 |
| 用户编辑你的输出 | 中 | 记录为临时模式 |
| 同一纠正出现3次 | 确认 | 询问是否永久化 |

### 3.2 不触发学习的场景

- ❌ 沉默（不是确认）
- ❌ 单次事件
- ❌ 假设性讨论
- ❌ 第三方偏好（"John喜欢..."）
- ❌ 群聊模式（除非用户确认）
- ❌ 推断的偏好

### 3.3 确认流程

```
纠正出现3次后：

Agent: "我注意到你更喜欢 X 而不是 Y（已纠正3次）。
       应该总是这样做吗？
       - 是的，总是
       - 只在 [特定场景]
       - 不，视情况而定"

User: "是的，总是"

Agent: → 移至"已确认偏好"
       → 重置纠正计数器
       → 后续使用时引用来源
```

### 3.4 模式演进阶段

```
Tentative (临时)     → 单次纠正，观察重复
    ↓
Emerging ( emerging) → 2次纠正，可能是模式
    ↓
Pending (待确认)     → 3次纠正，请求确认
    ↓
Confirmed (已确认)   → 用户批准，永久生效
    ↓
Archived (已归档)    → 90天未用，保留但不活跃
```

## 四、自我反思机制

### 4.1 触发时机
- 完成多步骤任务后
- 收到反馈（正面或负面）后
- 修复 bug 或错误后
- 发现自己的输出可以改进时

### 4.2 反思格式

```
CONTEXT: [任务类型]
REFLECTION: [我注意到了什么]
LESSON: [下次如何改进]
```

### 4.3 示例

```
CONTEXT: Flutter UI 构建
REFLECTION: 间距看起来不对，不得不重做
LESSON: 在展示给用户前，先截图检查视觉平衡
```

## 五、自动运维

### 5.1 会话启动时
1. 加载 memory.md（HOT层）
2. 检查 index.md 获取上下文提示
3. 如果检测到项目 → 预加载相关命名空间

### 5.2 收到纠正时
1. 解析纠正类型（偏好/模式/覆盖）
2. 检查是否重复（存在于任何层级）
3. 如果新的 → 添加到 corrections.md
4. 如果重复 → 增加计数器，计数器≥3时请求确认
5. 确定命名空间（global/domain/project）
6. 更新 index.md 行数统计

### 5.3 周维护（Cron）
1. 扫描所有文件的衰减候选
2. 将未使用>30天的移至 WARM
3. 将未使用>90天的归档至 COLD
4. 如果文件超限则执行压缩
5. 更新 index.md
6. 生成周报（可选）

## 六、安全边界

### 6.1 永不存储

| 类别 | 示例 | 风险 |
|------|------|------|
| 凭证 | 密码、API密钥、Token | 安全泄露 |
| 财务 | 银行卡、银行账户 | 欺诈风险 |
| 医疗 | 诊断、药物、病情 | 隐私/HIPAA |
| 生物特征 | 声纹、行为指纹 | 身份盗用 |
| 第三方信息 | 关于其他人的信息 | 无授权 |
| 位置模式 | 家庭/工作地址、行程 | 人身安全 |
| 访问模式 | 用户可访问的系统 | 权限提升 |

### 6.2 透明度要求
1. **按需审计** — 用户问"你知道我什么？"→ 完整导出
2. **来源追踪** — 每项都标记何时/如何学习
3. **解释行为** — "我做了X是因为你在[日期]说了Y"
4. **无隐藏状态** — 影响行为的必须可见
5. **删除验证** — 确认已移除，显示更新状态

### 6.3 紧急开关

用户说"忘记一切"：
1. 导出当前记忆到文件（供审查）
2. 清除所有学习数据
3. 确认："记忆已清除。重新开始。"
4. 不保留"幽灵模式"在行为中

## 七、扩缩容策略

### 7.1 规模阈值

| 规模 | 条目数 | 策略 |
|------|--------|------|
| 小型 | <100 | 单个 memory.md，无命名空间 |
| 中型 | 100-500 | 拆分到 domains/，基础索引 |
| 大型 | 500-2000 | 完整命名空间层级，积极压缩 |
| 海量 | >2000 | 按年归档，HOT层仅保留摘要 |

### 7.2 压缩规则

**合并相似纠正**：
```
BEFORE (3条):
- [02-01] 用Tab不用空格
- [02-03] Tab缩进
- [02-05] 请用Tab缩进

AFTER (1条):
- 缩进: Tab (确认3次, 02-01至02-05)
```

**精简冗长模式**：
```
BEFORE:
- 给Marcus写邮件时，用项目符号，保持在5条以内，
  不用术语，结论先行，他偏好早上发送

AFTER:
- Marcus邮件: 项目符号≤5, 无术语, BLUF, 上午优先
```

## 八、用户命令速查

| 命令 | 动作 |
|------|------|
| "你知道X什么？" | 搜索所有层级，返回匹配及来源 |
| "显示我的记忆" | 显示 memory.md 内容 |
| "显示[项目]模式" | 加载并显示特定命名空间 |
| "忘记X" | 从所有层级移除，确认删除 |
| "忘记一切" | 完全清除并提供导出选项 |
| "最近有什么变化？" | 显示最近20条纠正 |
| "导出记忆" | 生成可下载的归档 |
| "记忆状态" | 显示各层级大小、最后压缩时间、健康度 |

## 九、最佳实践

### 9.1 适用场景
- ✅ 长期项目协作
- ✅ 重复性任务
- ✅ 个性化工作流
- ✅ 多项目上下文切换

### 9.2 不适用场景
- ❌ 一次性任务
- ❌ 敏感信息处理
- ❌ 多用户共享账户（无用户隔离）

### 9.3 推荐配置
```markdown
## Self-Improving 模式

当前模式: Passive

可用模式:
- Passive: 仅从显式纠正中学习
- Active: 3次重复后建议模式
- Strict: 每个条目都需要确认
```
